## 2024-03-26 - Remote Code Execution via Scene Synchronization
**Vulnerability:** The scene synchronization RPC `update_node_property` in `scene_sync.gd` accepted any property modification from remote peers, including sensitive properties like `script` and `metadata/*`. Remote file synchronization in `file_sync.gd` and `scene_sync.gd` allowed peers to arbitrarily read and overwrite files in `res://addons/team_create` and `res://.godot`. Finally, `receive_project_settings` lacked any verification to ensure the settings changes originated from the authoritative server instance.

**Learning:** When building remote synchronization tools in Godot Engine, specifically those that use Godot's built-in MultiplayerAPI and RPCs to replicate Godot's node properties or filesystem state, you cannot trust the peers. Attackers could send an RPC to modify a node's script property, assign it a script containing malicious code (either via file sync or a base64 encoded byte array if it supported sub-resources like that), and achieve remote code execution (RCE) within the Godot editor or game instance of other connected clients. Additionally, allowing clients to overwrite the plugin's own source code in `res://addons/team_create` or configuration files in `res://.godot` provides another avenue for persistence and code execution. Server-authoritative sync messages like `project.godot` updates must strictly validate `multiplayer.get_remote_sender_id() == 1`.

**Prevention:**
1. **Property Allowlisting/Blocklisting:** Never blindly accept property updates via RPC. Maintain a strict allowlist of properties that are safe to sync (e.g., position, rotation, custom application logic variables), or blocklist highly sensitive properties like `script` and metadata properties that could be exploited.
2. **Directory Restrictions:** In file synchronization RPCs (both receiving and requesting), enforce strict bounds checking on paths. Disallow access to sensitive directories like `.godot/` (engine cache and configs) and `addons/` (plugin source code) unless absolutely required and heavily authenticated.
3. **Sender Verification:** Always verify the sender of administrative or global state updates (like project settings) using `multiplayer.get_remote_sender_id()` to ensure only the server (ID 1) can broadcast them.
